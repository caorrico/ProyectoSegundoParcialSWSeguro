\section{Introducción}
El paradigma \textit{Shift-Left Security} propone integrar controles de seguridad lo más temprano posible en el ciclo de desarrollo. Este proyecto materializa dicha visión mediante la creación de un ecosistema autónomo que utiliza **Minería de Datos Analítica** para predecir si un cambio en el código fuente (Pull Request) introduce riesgos de seguridad. La solución orquestada en GitHub Actions combina la velocidad del análisis estático con la profundidad del Machine Learning clásico, eliminando la dependencia de servicios externos costosos.

\section{Ingeniería de Datos y SEMMA}

\subsection{Fase SAMPLE: Consolidación de Datasets}
Se diseñó un repositorio de datos unificado que ingesta información de fuentes heterogéneas. La diversidad de lenguajes y contextos de vulnerabilidad es clave para la capacidad de generalización del modelo.

\begin{itemize}[noitemsep]
    \item \textbf{CodeXGLUE:} Provee fragmentos de código con anotaciones de defectos semánticos.
    \item \textbf{CVEFixes:} Extrae pares de código "vulnerable" y "corregido" directamente de repositorios vinculados a la base de datos nacional de vulnerabilidades (NVD).
    \item \textbf{D2A/MegaVul:} Aportan cientos de miles de funciones en C/C++ etiquetadas mediante análisis estático industrial.
    \item \textbf{Total de registros limpios:} 389,313.
\end{itemize}

\subsection{Fase EXPLORE: El desafío del Desbalance}
Durante la exploración, se determinó que solo el **8.1\%** de los fragmentos de código son vulnerables. Esta asimetría es natural en el desarrollo de software (el código seguro es la norma), pero requiere estrategias de entrenamiento específicas como el \textit{cost-sensitive learning} para evitar que el modelo ignore sistemáticamente la clase minoritaria.

\subsection{Fase MODIFY: Extracción de Características Avanzada}
La transformación de código fuente a vectores numéricos se realizó mediante tres extractores concurrentes, diseñados para capturar tanto la semántica léxica como la estructura sintáctica:

\begin{enumerate}
    \item \textbf{HashingVectorizer (Truco del Hashing):} Convierte fragmentos de código en vectores de alta dimensionalidad ($2^{18}$). A diferencia de un vocabulario estático que crece indefinidamente con cada nuevo nombre de variable, el hashing mapea tokens a un espacio fijo, permitiendo procesar cualquier base de código sin necesidad de pre-entrenar un diccionario gigante.
    \item \textbf{Análisis Estructural AST (Abstract Syntax Tree):} Mediante la integración de la librería industrial \textbf{tree-sitter}, el sistema descompone el código en su gramática formal. Se implementó un extractor robusto que identifica:
    \begin{itemize}
        \item \textbf{Pointer Declarators:} Conteo de nodos \texttt{pointer\_declarator} y \texttt{reference\_declarator}, fundamentales para identificar riesgos de gestión de memoria en C/C++.
        \item \textbf{Complexidad Estructural:} Profundidad máxima del árbol y conteo de nodos totales, que sirven como proxis de la mantenibilidad y probabilidad de errores ocultos.
        \item \textbf{Estructuras de Control:} Frecuencia de \texttt{if\_statement}, \texttt{for\_statement} y \texttt{method\_invocation} para modelar el flujo de ejecución.
    \end{itemize}
    \item \textbf{Extractor de Taint (Sources/Sinks):} Identifica mediante expresiones regulares optimizadas la presencia de funciones de entrada (\textit{Sources}) como \texttt{scanf, argv, getenv} y funciones de destino peligrosas (\textit{Sinks}) como \texttt{strcpy, system, memcpy}. La coexistencia de ambos en un mismo fragmento eleva drásticamente el peso de la predicción de vulnerabilidad.
\end{enumerate}

\section{Modelado y Evaluación (Fase MODEL/ASSESS)}

\subsection{Justificación Algorítmica y Estrategia de Pesos}
Se seleccionó **XGBoost** (Extreme Gradient Boosting) como clasificador principal por su excepcional desempeño en datos tabulares y su capacidad nativa para manejar el desbalance de clases mediante el parámetro \texttt{scale\_pos\_weight}. 

La estrategia se basó en el \textbf{Cost-Sensitive Learning}: al asignar un peso mayor a la clase minoritaria ($11.4$), el algoritmo penaliza más severamente el falso negativo (dejar pasar una vulnerabilidad) que el falso positivo. Esto es crítico en seguridad, donde el costo de una brecha es infinitamente superior al de una revisión manual adicional.

\subsection{Explicabilidad mediante SHAP}
Para superar la naturaleza de "caja negra" de los modelos de boosting, se integró un motor de explicabilidad basado en \textbf{SHAP (SHapley Additive exPlanations)}. Esto permite descomponer cada predicción en la contribución individual de cada característica. En el pipeline DevSecOps, esto se traduce en mensajes de retroalimentación útiles para el desarrollador, indicando exactamente qué patrones de código dispararon la alerta de seguridad.

\section{Ecosistema DevSecOps: Orquestación e Infraestructura}

El pipeline de seguridad implementado en \textbf{GitHub Actions} actúa como el sistema nervioso del proyecto, orquestando múltiples etapas de validación:

\begin{enumerate}
    \item \textbf{Status Checks y Reglas de Protección:} El resultado de la inferencia de IA se reporta directamente a la API de GitHub como un chequeo de estado obligatorio. Si el modelo identifica un riesgo alto, el botón de "Merge" se bloquea, forzando una revisión manual.
    \item \textbf{Gestión Automática de Vulnerabilidades:} Ante una detección positiva, el pipeline:
    \begin{itemize}
        \item Crea un \textbf{GitHub Issue} con el label \texttt{fixing-required}, vinculándolo automáticamente al Pull Request original.
        \item Publica un comentario detallado en el PR con recomendaciones específicas por CWE y los resultados de la explicabilidad del modelo.
    \end{itemize}
    \item \textbf{Notificaciones Multi-Canal (Telegram):} Se implementó un bot de Telegram que consume la API de mensajes para notificar instantáneamente al equipo sobre el estado de cada etapa (Inferencia, Pruebas Unitarias y Despliegue en Render).
\end{enumerate}

\subsection{Contenerización y Despliegue}
Para asegurar la reproducibilidad del entorno de ejecución, se diseñó un \textbf{Dockerfile} multietapa que empaqueta todas las dependencias (Python, Tree-Sitter, XGBoost). Esto permite que la API de inferencia pueda desplegarse de forma elástica en plataformas como \textbf{Render}, garantizando que el servicio de escaneo esté siempre disponible y aislado de conflictos de dependencias locales.

\section{Conclusiones}
La solución implementada demuestra un equilibrio entre rigor analítico y operatividad DevSecOps. El uso de la metodología SEMMA permitió pasar de una recolección de datos masiva a un sistema de decisión distribuido que educa al desarrollador en lugar de solo castigar el error. La integración de la ingeniería de características (AST + Hashing) dota al modelo de una visión semántica del código que supera el análisis de texto simple.
