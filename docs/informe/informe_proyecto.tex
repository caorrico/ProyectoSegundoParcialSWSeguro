\section*{Resumen}
El proyecto implementa un flujo DevSecOps para clasificar cambios de código como
\textit{SAFE} o \textit{VULNERABLE} sin usar modelos LLM. La solución combina limpieza de
datasets, entrenamiento de un modelo clásico de minería de datos, análisis de Pull Requests,
notificaciones por Telegram, bloqueo de merge ante vulnerabilidades, ejecución de pruebas y
despliegue automatizado. El notebook completo fue ajustado para entrenar con todos los
registros disponibles, usando una arquitectura escalable basada en hashing y clasificación
lineal.

\section{Objetivo}
Construir un pipeline seguro que analice código modificado en Pull Requests, detecte patrones
vulnerables y automatice acciones de seguridad: comentario en PR, etiquetado, creación de issue,
notificación por Telegram, pruebas, merge y despliegue. Esta aproximación se alinea con prácticas
DevSecOps y con la automatización soportada por GitHub Actions \cite{githubactions}.

\section{Trabajo realizado}
Las actividades principales fueron:

\begin{itemize}
    \item Limpieza y unificación de datasets de vulnerabilidades: CodeXGLUE, D2A, ReVeal,
    MegaVul, CVEFixes y OWASP 2025.
    \item Segmentación por lenguaje mediante la columna \texttt{language}.
    \item Entrenamiento del notebook completo con todos los registros: \textbf{389313}
    muestras limpias.
    \item Uso de \texttt{HashingVectorizer} y \texttt{SGDClassifier} para escalar el
    entrenamiento sin construir vocabularios gigantes \cite{sklearnhashing,sklearnsgd}.
    \item Implementación de análisis de PR con reglas, probabilidad del modelo y reporte JSON.
    \item Configuración de Telegram por secrets: \texttt{TELEGRAM\_BOT\_TOKEN} y
    \texttt{TELEGRAM\_CHAT\_ID}, usando la API HTTP oficial \cite{telegramapi}.
    \item Actualización de workflows de GitHub Actions para lint, tests, escaneo, bloqueo y deploy.
\end{itemize}

\section{Arquitectura de la solución}

La arquitectura del pipeline integra una etapa de análisis estático heurístico (basado en expresiones regulares para detectar patrones de llamadas a funciones peligrosas como \texttt{strcpy}, \texttt{gets}, \texttt{eval}, etc.) antes de la inferencia del modelo de Machine Learning, asegurando una detección rápida de vulnerabilidades conocidas.

\section{Análisis Estático y Entrenamiento}
El pre-análisis heurístico utiliza expresiones regulares definidas en \texttt{scripts/ci\_inference\_pipeline.py} para identificar llamadas a funciones peligrosas (vulnerabilidades de desbordamiento de búfer e inyección). El entrenamiento del modelo se realiza siguiendo un proceso de carga de datos, limpieza, extracción de características (AST y patrones de seguridad), y clasificación mediante \texttt{XGBClassifier} optimizado con \textit{scale\_pos\_weight} para manejar el desbalance de clases.\par

\subsection{Modelo escalable del notebook completo}
\begin{lstlisting}[style=pythonstyle, language=Python]
features = ColumnTransformer([
    ('word_hash',
     HashingVectorizer(n_features=2**18,
                       ngram_range=(1, 2),
                       alternate_sign=False,
                       norm='l2'),
     'raw_code'),
    ('category',
     OneHotEncoder(handle_unknown='ignore', sparse_output=True),
     ['language', 'source']),
    ('numeric',
     StandardScaler(with_mean=False),
     ['code_length', 'line_count']),
])
\end{lstlisting}

\noindent
Aunque el accuracy supera el 80\%, se reportó también el \textit{baseline} de clase mayoritaria
(0.9228), ya que el dataset completo está desbalanceado. Por ello, ROC AUC, recall y F1 de la
clase vulnerable son métricas más informativas que accuracy por sí sola.

\section{Conclusiones}
Se dejó un repositorio funcional con notebooks reproducibles, pipeline CI/CD seguro y análisis
automatizado de Pull Requests. El entrenamiento completo ya usa todos los registros disponibles y
el pipeline puede bloquear código vulnerable identificado por reglas o por alta probabilidad del
modelo. Como mejora futura, se recomienda entrenar modelos especializados por lenguaje, en
especial para C/C++ y Java, para mejorar el recall de vulnerabilidades sin aumentar falsos
positivos.
